Principal Engineer Control Tower

A local-first engineering control plane for eligible PR review, evidence-backed delegated review, and independently evaluated delivery capabilities.

Deterministic coverage and authority; agentic judgment and advice.
PHASE MODEL Phase 1 is the baseline. Phase 2 is a menu of four parallel, independently gated capabilities—2A advanced review, 2B bot publication, 2C delivery intelligence, and 2D sandboxed checks. It is not a sequence, and no capability is required to ship.
System boundary

Coverage is computed. Judgment is delegated. Authority stays human.

The application supplies facts, constraints, identities, state, credentials, and publisher guards. Cursor roles provide bounded analysis. A principal engineer explicitly starts optional work and approves each external operation.

Deterministic control plane APPLICATION

  • GitHub/provider acquisition and eligible-review cache
  • Eligibility, priority, auto-analysis, domains, triggers
  • Canonical path filtering and protected-byte exclusion
  • Application-created provenance/facts and validation
  • Immutable identities, attempts, state, currentness
  • Credential boundaries and publisher/executor guards

Cursor roles ADVICE

  • Primary evidence-backed review
  • Specialist review advice and analysis when 2A allows
  • Delivery interpretation when 2C allows

Human authority PRINCIPAL

  • Manual analysis, specialist, and sandbox starts
  • Explicit reduced-related-source decisions
  • Exact, single-use approval per external operation
  • Capability selection, gates, pilots, acceptance, disable

Advice never becomes authority

Agent confidence, relevance, risk, recommended action, and recommended disposition are context for the human. They cannot make a PR eligible, enqueue work, pass a gate, execute a check, publish, or advance autonomy.

Proposed baseline contract

Phase 1 · delegated PR review

CI-first and local-first. It persists only eligible PRs, executes one primary review role, validates all evidence, and publishes only through exact principal approval.

Eligibility and review queue

Only eligible PRs are persisted in the review-core database. Ineligible PR metadata is not durable product state.

Eligible review cache is authoritative

Eligible open PRs and explicit review requests to the configured operator are persisted with a policy snapshot for Inbox ordering. Non-reviewable PRs are not written to the review queue.

explicit request to configured operator OR (active repository AND (eligible path OR exact normalized eligible author))
  • Every explicit request to the configured operator is eligible—even inactive or unregistered, when readable.
  • Author matching is exact normalized GitHub login equality, never display name, team, prefix, or substring.
  • Every match or exclusion reason is stored.

Priority and execution are separate

p0 > p1 > p2 > p3 > unranked

Every eligible PR defaults to p3. Ineligible PRs are not persisted. Author-only eligibility is on-demand unless another deterministic auto-analysis rule independently matches.

Focus Queue · eligible only

  • Now: at most three highest-priority actionable items.
  • Next: other eligible items, including on-demand candidates.
  • Monitor: eligible items temporarily non-actionable.

Deterministic order is default.

Review pipeline

Registered source and remote evidence are explicit modes. A registered-source failure never silently degrades.

01 DISCOVERGitHub metadata, changed paths, requests, checks
02 DECIDEEligibility, tier, auto-analysis, domains, source mode
03 FILTERStream GitHub diff; discard protected bytes before every sink
04 MATERIALIZEAuthenticated fetch, then credential-free direct regular blobs
05 REVIEWOne primaryReview Cursor attempt with frozen inputs
06 VALIDATESchema, coverage, provenance, inline locations, model identity

Registered filtered source

Authenticated partial-mirror fetch ends before credential-free SHA verification and materialization. A no-checkout admin worktree handles lifecycle only. Cursor receives a separate read-only source view created directly from allowed regular blobs.

Excluded: protected paths, symlinks, gitlinks/submodules, special/unknown modes, unsafe or colliding paths, Git metadata, hooks, attributes, filters, and repository commands.

Remote evidence only

For an explicit request outside registered source: filtered GitHub metadata, diff, commits, discussion, reviews, and checks. It creates no mirror, admin worktree, source view, --add-dir, or file provenance.

sourceTreeInspected: false and missingCoverage: ["source_tree"] remain visible.

Failure stays explicit

A registered-source fetch, SHA, tree, object, or materialization failure fails the source-backed job and stays visible. A human may explicitly start a new remote-evidence-only job; the system never performs that downgrade automatically.

Evidence, attempts, and workbench

Evidence is minted by application code; agents can cite it or submit exact file locators for deterministic validation.

Application provenance

pv_ IDs bind canonical records for filtered diff hunks, checks, comments/reviews, and commits. Agents cite IDs; they never mint or redefine them.

File claims must validate exact repository, blob SHA, canonical path, and inclusive line range against the allowed source manifest.

Immutable attempts

Jobs have immutable run attempts and create-once inputs, append-only transcripts, sealed terminal records, and transactional latest/accepted pointers. Manual retry creates a new attempt. Input changes supersede prior drafts; restart failures stay visible.

Understand · Verify · Act

The workbench separates intent/scope/checks, provenance/findings/unknowns, and editable structured operations. It always warns: CI results were observed; local checks were not run.

Remote evidence and protected/filter omissions add persistent coverage warnings.

Independent capability menu

Phase 2 · four parallel evaluation lanes

After relevant Phase 1 gates pass, any selected lane may proceed on its own. Accepted, deferred, rejected, disabled, and not evaluated are all valid independent outcomes.

No sequential roadmap and no aggregate completion flag

2A–2D do not depend on one another unless a future approved design explicitly says so. Every capability owns its plan, branch, migrations, feature flag, declared baseline, canonical identity, evaluation epoch, gates, pilot, acceptance review, and disable path.

Contract change → new identity

Any contract-affecting config, scope, role, harness, schema, safety, provenance, credential-state, binding, migration, or declared-baseline change creates a new cei_ identity in not_evaluated. Prior authority does not carry.

Continuing gate loss → new epoch

For the same identity, gate loss fences authority, archives old evidence, clears pilot/acceptance/current pointers, increments the durable evaluation epoch and scheduler fence, creates a new ceac_, and requires every gate again.

Autonomy ladder

Level 2 is the highest level authorized by these draft designs.

L0 · Shadow

Predictions retained for evaluation only; not actionable or externally visible.

L1 · Advice

Recommendations and drafts visible to humans; no external mutation.

L2 · Exact approval

One exact per-action human approval, then deterministic revalidation and one operation.

L3 · Future design

Not implemented. Requires a separate approved design; readiness telemetry grants no authority.

Outside the ladder

No autonomous approve, request-changes, merge, principal disposition, Linear write, arbitrary command, or silent learning.

No agent credentials

No analysis role receives GitHub, SSH, Git credential-helper, Linear, bot, registry, cloud, package, repository, or sandbox credentials. Each deterministic child environment is constructed independently and minimally.

Canonical path and protected-byte rule

One matcher contract

A canonical repository path is root-relative, case-preserving, valid UTF-8/NFC, /-separated, and free of leading/trailing slash, backslash, controls, empty/./.., or case-insensitive .git segments.

One compiled, content-hashed CanonicalPathMatcher serves eligibility, priority, domains, protected diff/source, capability paths, provenance, and fail-closed reads.

Filter before every sink

Protected diff bytes are discarded by the streaming parser before disk, logs, SQLite, artifacts, provenance, browser, or agent context. Protected source blobs are never requested. Only canonical path names and explicit omission reasons survive.

Malformed/ambiguous filtering fails closed and produces visible missing coverage.

Provenance and fact flow

01 ACQUIREDeterministic adapters collect exact provider/GitHub/Git observations
02 FILTERCanonicalize paths and exclude protected/unsafe content
03 CREATEApplication mints immutable pv_, src_, and fact_ records
04 FREEZEBounded catalogs enter immutable run input with hashes and coverage
05 CITEAgent cites supplied IDs or submits exact file locator
06 VALIDATEApplication verifies identity, currentness, repository/blob/path/range, and schema

Body-bearing publication requires evidence

Inline comments, summary comments, COMMENT reviews, REQUEST_CHANGES reviews, and bot comments require non-empty current application-validated citations. Only bodyless principal APPROVE may have empty provenance.

Safe browser rendering

Loopback boundary

The product binds only to loopback. It rejects non-loopback host headers and cross-origin requests. State changes require same-origin checks and a short-lived, single-use action token created by an explicit gesture.

Untrusted content

PR, provider, specialist, bot, sandbox, and agent text is untrusted. Plain fields use text nodes. Markdown has raw HTML off, a strict sanitizer, and only relative/fragment/https:/mailto: URLs.

Defense in depth

Restrictive CSP; no eval/Function; no runtime-generated script/style URLs or controls from untrusted content; no forms, iframes, objects, SVG, script, style, or active media.

Sandbox boundary and terminal failures

Execution envelope

  • Prepared digest-pinned image and verified immutable supervisor/collector
  • Read-only container root and filtered source
  • One quota-bounded tmpfs scratch mount; no other writable volume
  • No network, credentials, shell, Docker socket, host PID/IPC/devices
  • Unprivileged child, dropped capabilities, no-new-privileges, private PID namespace and controlled cgroup

Never convert failure to pass

  • resource_exhausted for byte/inode exhaustion—even after child exit zero
  • collector_failed: descendants_not_quiesced when process table and cgroup do not prove stable emptiness
  • collector_failed for supervisor, collector, channel, frame, or safe traversal failure
  • timed_out, unavailable, cancelled, or failed remain explicit

Collection occurs against live, untrusted-immutable scratch with descriptor-relative no-follow traversal before teardown.

Readable identity and invalidation model

dbe_ · declared baseline

A content hash of one immutable, acyclic baseline payload. It binds the delivered Phase 1 contract, a sealed prior accepted capability, or approved external evidence—plus corpus/evidence and exact metric-definition schema. It must not depend on the current capability identity, epoch, attempt context, output, or envelope.

cei_ · canonical capability identity

Hashes exact capability implementation/config/scope, enabled role specs, fixed evaluation harness manifests, schemas, safety/provenance contracts, non-secret credential permission state, capability bindings, migrations, and the dbe_ baseline.

Evaluation epoch

A durable positive integer outside cei_. A same-contract evaluation restart or continuing gate loss increments it. Returning to identical bytes never restores prior authority.

ceac_ · attempt context

A hash of cei_ + evaluationEpoch. Every gate, job, output, approval, and status freezes both. A separate scheduler epoch fences leases during transitions.

Invalidation is capability-local and atomic

A contract change moves only that capability to a new cei_ in not_evaluated. Continuing gate loss keeps the identity but creates a new epoch. In both cases authority-bearing pointers are cleared before new work can lease; old jobs may remain immutable history but cannot satisfy or restore new authority. Unaffected capabilities remain unchanged.

Valid independent outcomes

Accepted means measured value and safety for one identity/epoch/scope. Deferred means no qualifying current need. Rejected means a gate failed. Disabled means not selected. Each is honest and valid independently; there is no aggregate “Phase 2 complete.”
